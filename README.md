# Deckboss

> Your AI assistant for edge robotics and IoT systems. Clone this repo onto a Jetson or Raspberry Pi, run one command, and start talking to your hardware.

**The build-phase companion.** Design systems, vibe-code solutions, then hand off to Cocapn for deployment.

## Quick Start (Jetson Super Orin Nano)

```bash
# 1. Clone the repo
git clone https://github.com/Lucineer/deckboss.git
cd deckboss

# 2. Run setup (installs Python deps, prompts for configuration)
bash setup.sh

# 3. Start Deckboss
deckboss

# That's it. Deckboss boots, walks you through onboarding,
# and you're talking to your Jetson.
```

## What Happens on First Run

Deckboss runs an onboarding wizard:

1. **API Keys** — Enter your keys for cloud LLM providers (DeepSeek, z.ai, DeepInfra, SiliconFlow). Or skip and use local models only.
2. **Local Models** — Choose your inference engine:
   - **Ollama** (recommended) — serial single-model use, easy setup, `ollama pull phi3:mini`
   - **vLLM** — parallel/swarm inference, better for multi-agent workloads, more RAM needed
3. **Input/Output** — Set up how you talk to Deckboss:
   - Terminal (default) — text-only in the console
   - Telegram bot — talk from your phone
   - Discord bot — talk from Discord
   - LAN API — serve an HTTP endpoint on your local network
   - Google Docs sync — monitor folder changes
4. **Hardware Detection** — Deckboss scans for connected devices:
   - CSI cameras (Jetson ribbon cameras)
   - USB cameras (webcams, thermal)
   - USB microphones/speakers (for STT/TTS)
   - GPIO pins (RPi sensor connections)
5. **Profile Selection** — Choose which git-agent profile to equip:
   - **Lucineer/marine** (default) — IoT in marine environments, local-first edge
   - Or add custom profiles from other developers

After onboarding, Deckboss creates a **character sheet** — a resource allocation plan that determines how your Jetson's memory, compute, and I/O are arranged based on your hardware and chosen role.

## The Character Sheet

Deckboss generates a `~/.deckboss/character.yaml` file that defines:

```yaml
# Auto-generated by Deckboss onboarding
hardware:
  device: "Jetson Super Orin Nano"
  ram_gb: 8
  vram_gb: 6  # shared GPU/CPU
  storage: "/mnt/nvme"
  detected:
    csi_cameras: 2      # global shutter cameras for robotics
    usb_microphone: null
    usb_speaker: null
    gpio_available: false

resource_plan:
  model_engine: "ollama"   # ollama for serial, vllm for parallel
  max_gpu_model_size_gb: 4 # leave headroom for OS + inference
  pipeline_mode: "serial"  # serial: one model at a time
  # If 2 CSI cameras were detected:
  # pipeline_mode: "parallel"  # cameras need concurrent inference
  # max_gpu_model_size_gb: 3   # tighter budget for parallel

models:
  stt:
    engine: "whisper"
    source: "local"
    model: "medium"
    priority: 3  # loaded on demand
  tts:
    engine: "piper"
    source: "local"
    model: "en_US-lessac-medium"
    priority: 5  # always available (tiny)
  primary:
    engine: "ollama"
    source: "local"
    model: "phi3:mini"
    context: 4096
    priority: 1  # always available
  reasoning:
    engine: "cloud"
    provider: "deepseek"
    model: "deepseek-reasoner"
    api_key_env: "DEEPSEEK_API_KEY"
    priority: 2  # loaded when internet available

io:
  primary: "terminal"
  secondary: []
  lan_api:
    enabled: false
    port: 8080
  telegram:
    enabled: false
    bot_token_env: "TELEGRAM_BOT_TOKEN"
  discord:
    enabled: false
    bot_token_env: "DISCORD_BOT_TOKEN"

profile: "lucineer/marine"

agents_path: "~/.deckboss/agents"
logs_path: "~/.deckboss/logs"
```

**Why this matters:** A Jetson with 2 CSI cameras for global-shutter stereo vision in robotics arranges resources completely differently than a Cocapn acting as an ideation agent that expands repos all night. The character sheet makes these tradeoffs explicit and configurable.

## Example Setups

### Marine Technician (Your Setup)
```bash
deckboss onboard --profile lucineer/marine
# Detected: Jetson Orin Nano 8GB, no cameras, no mic
# Models: Whisper (STT via phone), Phi-3-mini (local), DeepSeek (cloud)
# I/O: Terminal + Telegram (phone on boat)
# Role: System designer — vibe-code marine IoT installations
```

### Robotics Lab
```bash
deckboss onboard --profile custom/robotics
# Detected: Jetson Orin Nano 8GB, 2x CSI cameras, USB mic
# Models: Ollama Phi-3 (local), vLLM for parallel camera inference
# I/O: Terminal + LAN API (robot control endpoint)
# Role: Real-time vision and motor control
```

### Content Creator (Pure Software)
```bash
deckboss onboard --profile custom/content
# Detected: Raspberry Pi 5 8GB, no cameras
# Models: Cloud-only (DeepSeek, z.ai) — Pi handles orchestration
# I/O: Terminal + Discord
# Role: Content pipeline agent — manages posts, replies, analytics
```

### Night Owl Developer
```bash
deckboss onboard --profile lucineer/marine --role ideation
# Detected: Jetson Orin Nano 8GB, no cameras
# Models: DeepSeek-Reasoner (cloud), DeepSeek-chat (bulk work)
# I/O: Terminal only (overnight unattended sessions)
# Role: Fleet expansion — spawn agents, fill capability gaps, PR all night
```

### Vibe-Coding Session

Deckboss uses a Claude Code-like interaction model for system design:

1. You describe what you need in plain language
2. A higher-tier model (usually cloud, DeepSeek-Reasoner) creates the design
3. Deckboss proposes a git-agent fleet — one repo per subsystem
4. You riff on the design — "move that sensor here", "add redundancy"
5. Deckboss updates the character sheet and generates implementation code
6. When ready, hand off to Cocapn: `deckboss handoff --to cocapn`

### Managing Agents

```bash
# List equipped agents
deckboss agents

# Pull a new agent from a profile
deckboss pull lucineer/marine/engine-monitor

# Start an agent session
deckboss session engine-monitor

# Check resource usage
deckboss status

# Edit character sheet
deckboss config edit
```

## Git-Agent Architecture

Every agent is a git repo in `~/.deckboss/agents/`. The repo IS the agent:

```
~/.deckboss/agents/
└── engine-monitor/
    ├── README.md           # Agent description and capabilities
    ├── agent.yaml          # Agent configuration (inputs, outputs, model prefs)
    ├── main.py             # Agent runtime
    ├── skills/             # What the agent knows how to do
    │   ├── gauge_reading.md
    │   ├── flood_detection.md
    │   └── temp_alerting.md
    └── equipment/          # What the agent perceives through
        └── sensors.yaml    # Connected sensor definitions
```

### Profiles

Profiles are curated collections of git-agent repos. Add profiles:

```bash
# Add a profile source
deckboss profiles add lucineer https://github.com/Lucineer

# List available profiles
deckboss profiles list

# Browse agents in a profile
deckboss profiles browse lucineer/marine
```

Anyone can create a profile. A robotics developer could create `acme/warehouse` with agents for inventory tracking, forklift monitoring, and safety compliance. Deckboss users add that profile and equip those agents on their hardware.

### Hand Off to Cocapn

When your design is complete:

```bash
deckboss handoff --to cocapn
```

This packages your agent fleet into a Cocapn-ready format:
- Generates `cocapn.yaml` for each agent
- Configures runtime bindings
- Sets up monitoring pipelines
- Creates installation documentation
- Deploys agents as services on the Jetson

Deckboss powers down to cold storage. Cocapn takes over.

## Requirements

- Python 3.10+
- Ubuntu (tested on NVIDIA JetPack 6.x Ubuntu, Raspberry Pi OS)
- Git
- 8GB+ RAM recommended (4GB minimum)
- Optional: NVIDIA GPU (CUDA 12.x for Jetson, CUDA 12.x for desktop)

## Install From Scratch (Fresh Jetson)

```bash
# Flash JetPack 6.x to NVMe (follow NVIDIA docs)
# Boot, SSH in, then:

sudo apt update && sudo apt install -y python3 python3-pip git
git clone https://github.com/Lucineer/deckboss.git
cd deckboss
bash setup.sh
deckboss
```

## Philosophy

- **The repo IS the agent.** Every capability, every decision, every evolution lives in git.
- **Hardware-aware.** A Jetson with cameras arranges resources differently than a Pi running cloud-only.
- **Local first.** Works without internet. Cloud is enhancement, not requirement.
- **Profile-driven.** Marine IoT, robotics, content creation — same architecture, different agents.
- **BYOK everywhere.** Your API keys, your models, your data. Zero lock-in.
- **Under-promise, over-deliver.** Everything in the Lucineer ecosystem.
